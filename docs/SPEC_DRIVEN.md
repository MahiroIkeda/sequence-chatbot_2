# 仕様駆動開発（Specification-Driven Development）について

本ドキュメントは、SpecFlowが基づく「仕様駆動開発」の理論的背景と、本ツールの各機能がどの原則・工程に対応しているかを解説する。

---

## 仕様駆動開発とは

「仕様そのものを出発点・原動力としてシステム開発全体を推進する手法」である。

従来、仕様書はプロジェクト開始時に作成され、その後は更新されないまま放置されることが多かった。コードと乖離した仕様書、誰も読まない設計書——これが「仕様の形骸化」と呼ばれる問題である。AI技術の登場により、書いた仕様からコードを自動生成できるようになったことで、仕様を開発の原動力とすることが初めて現実的になった。

仕様駆動開発は、ウォーターフォール開発の「仕様の明確化」という強みと、アジャイル開発の「変化への柔軟な対応」という強みの両方を取り入れた開発方法論である。開発プロセス（ウォーターフォール／アジャイル）とは独立した概念であり、どちらのプロセスにも適用できる。

また、「仕様駆動開発（Specification-Driven Development）」という言葉自体は、現在の生成AIが登場するはるか前の2004年に論文として登場しており、近年のAIブームで生まれたバズワードではない。

---

## Spec-as-Code という考え方

仕様駆動開発の基本概念として **Spec-as-Code** がある。仕様をコードと同じように管理・運用する考え方である。

具体的には、仕様を Markdown で記述し、Git/GitHub で管理し、AIで活用することそのものを指す。仕様を機械可読な形式で記述することで、AIが処理でき、継続的に更新できる「生きたドキュメント」として機能する。

SpecFlow はこの Spec-as-Code を実践するためのツールとして設計されている。

---

## 4つの原則

### 原則1 — 仕様は"生きたドキュメント"

仕様は一度書いて終わりではなく、継続的に更新される動的な成果物として扱う。実装で得られた学びを仕様に反映し、常に最新のプロジェクト状況を示す状態を保つ。

「生きたドキュメント」は「いつでも自由に変えていい」という意味ではない。変更は必ず履歴として残し、関係者に共有し、影響範囲を分析することが前提となる。「動的」であることと「無秩序」は全く異なる。

**SpecFlowでの対応**
GitHubへの保存のたびに `docs/specification.md` をコミットし、変更履歴を蓄積する。SpecFlowを通じて保存するたびに、仕様書・シーケンス図・READMEが自動更新される。

---

### 原則2 — 仕様は"信頼できる唯一の情報源（SSOT）"

会話やメール、散逸した資料に頼らず、すべてのメンバーが「仕様」に立ち返る。迷ったときは仕様に戻り、そこを出発点に修正や意思決定を行う。知識の属人化——「あの機能を知っているのは田中さんだけ」という状況——を防ぐために、仕様をGitHub上でチーム全員がアクセスできる場所に置く。

**SpecFlowでの対応**
GitHubリポジトリの `docs/specification.md` を唯一の参照先として管理する。差分検出機能により「今の仕様とGitHub上の仕様がどう違うか」を常に把握できる。

---

### 原則3 — 仕様は"変更と反復が前提"

変化を例外扱いするのではなく、必然として取り込み続ける。「何が、なぜ変わったか」をGitHubのコミット履歴として可視化し、各サイクルで仕様を見直す機会を設ける。変更を「失敗」ではなく「学びの反映」として記録する文化を育てる。

「変更と反復が前提」は「計画不要」という意味ではない。むしろ、変更が起きることを前提に計画するからこそ、より現実的で柔軟な計画が立てられる。

**SpecFlowでの対応**
Pull Request機能により、仕様変更をブランチ単位で追跡・承認できる。差分検出でAIが変更点と影響範囲を解説する。

---

### 原則4 — AIでコストを抑える

仕様を「信頼できる唯一の情報源」として入力し、AIの支援で生産性を向上させ、仕様の更新コストを削減する。「AIに丸投げする」という意味ではない。「何を作るか」を定義するのは人間の責任であり、最終判断は人間が行う。AIはその実現を支援するものとして位置づける。

**SpecFlowでの対応**
曖昧性検出・仕様レビュー・シーケンス図生成・PR説明文の作成まで Claude API で自動化する。人間はレビューと承認を担う（Human-in-the-Loop）。

---

## 7つの工程

仕様駆動開発は以下の7工程で構成される。各工程は一方向に進むのではなく、必要に応じて反復・行き来することが前提である。

```
1. 原則決定（Constitution）
2. 企画・要件定義（Specify）
3. 設計計画（Plan）
4. タスク分割（Tasks）
5. 実装（Implement）
6. 検証・受入（Verify/Accept）
7. 移行・運用（Migration/Operation）
```

工程間の反復例：
- 実装中に仕様を更新（工程5 → 工程2）
- 検証で問題が見つかれば設計に戻る（工程6 → 工程3）
- 運用中の学びを次の企画に活かす（工程7 → 工程2）

---

### 工程1 — 原則決定（Constitution）

プロジェクトの存在意義・基本原則・制約条件を明文化し、全ステークホルダーの合意を形成する工程。PMBOKでいう「プロジェクト憲章（Project Charter）」に相当する。`Constitution.md` として管理し、以後のすべての工程における「前提」として機能する。

仕様変更が提案された場合、「それはプロジェクト憲章の基本原則に沿っているか」を確認する。原則の変更は例外的・特別な扱いとし、仕様や設計の変更とは明確に区別される。

**SpecFlowでの対応**
本リポジトリの README.md および本ドキュメントが、プロジェクト憲章に相当する。

---

### 工程2 — 企画・要件定義（Specify）

「何を」「なぜ」作るのかを定義する工程。従来の要件定義書のように最初に完璧を目指すのではなく、「進化する仕様」として扱う。最初は抽象度の高い要求から始め、段階的に詳細化していく。

重要なのは、分類の形式ではなく「仕様が明確であること」「チーム全員が理解できること」「AIが処理できる形式（Markdown）で書かれていること」である。

**SpecFlowでの対応**
アップロードした要件定義書が工程2の成果物に相当する。「曖昧性を自動検出」機能が、AIの視点で仕様の不明瞭な箇所を特定・補完する。

---

### 工程3 — 設計計画（Plan）

技術スタック・アーキテクチャ・制約条件を整理し、設計プランを策定する工程。複数の設計案を比較検討し、チーム全体で合意形成するプロセスが重要となる。AIの提案をそのまま採用するのではなく、必ずレビューを設ける。

**SpecFlowでの対応**
「仕様をレビュー」機能が、明確性・完全性・一貫性・実装可能性の4観点で仕様を分析し、設計計画の土台を整える。

---

### 工程4 — タスク分割（Tasks）

仕様と設計をもとに、レビュー・テスト可能な粒度でタスクを分割する工程。「1タスク＝レビューが30分以内に終わる大きさ」といった具体的なガイドラインを設けることで、レビュー負荷を適切に抑えられる。

**SpecFlowでの対応**
シーケンス図を生成した後、Mermaidコードを直接編集してタスク単位での実装範囲を整理できる。

---

### 工程5 — 実装（Implement）

タスクごとにコードを書き、人間がレビュー・検証する工程。「人間が実装すべき内容を書く」ことから始まり、「AIが生成したコードを人間が読む」ことによって終わるサンドイッチ構造（Human-in-the-Loop）を明示的に組み込む。

1つのタスクが完了したらすぐにコミットし、小さな単位で逐次確認することで、課題の早期発見が可能になる。

**SpecFlowでの対応**
シーケンス図のチャット修正機能が、実装中に発生した仕様変更をインタラクティブに反映する手段となる。

---

### 工程6 — 検証・受入（Verify/Accept）

実装された機能が仕様を満たしているかを確認する工程。「最後にまとめて行う」のではなく、各スプリント・各実装タスクで継続的に実施する。仕様駆動開発では、プロジェクト開始時の仕様と現在の仕様の差分をレポートとして可視化し、顧客に説明する。

**SpecFlowでの対応**
「差分を検出」機能が、前回GitHubに保存した仕様と現在の仕様を比較し、AIが変更点・影響範囲・推奨アクションを解説する。

---

### 工程7 — 移行・運用（Migration/Operation）

システムを本番環境へ移行し、運用フェーズで仕様変更や改善を継続的に反映する工程。運用フェーズは終点ではなく、新たな学びの始点である。運用中に発見されたバグ・ユーザーからの改善要望・市場環境の変化は、すべて仕様に反映され、工程1または工程2に戻り新たなサイクルが始まる。

**SpecFlowでの対応**
Pull Request機能により、運用中の仕様変更をブランチ単位で管理し、承認フローを経てmainブランチに反映できる。

---

## SpecFlowと7工程の対応まとめ

| 工程 | SpecFlowの機能 |
|------|---------------|
| 1. 原則決定 | README.md・本ドキュメント |
| 2. 企画・要件定義 | 要件定義書アップロード・曖昧性を自動検出 |
| 3. 設計計画 | 仕様をレビュー |
| 4. タスク分割 | シーケンス図の編集・整理 |
| 5. 実装 | チャットによる図の修正・反映 |
| 6. 検証・受入 | 差分を検出 |
| 7. 移行・運用 | GitHubに保存・Pull Request作成 |
